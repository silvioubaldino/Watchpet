Visão Geral do Produto 1.1 Conceito WatchPet é um aplicativo nativo para Apple Watch que combina um modelo leve de linguagem (LLM) com transcrição de áudio em tempo real para criar uma experiência de companheiro virtual — um &quot;pet digital&quot; — diretamente no pulso do usuário. O app transcende a categoria de assistentes tradicionais ao oferecer uma personalidade persistente, memória contextual, lembretes proativos inteligentes e um bloco de notas ativado por voz, tudo funcionando de forma fluida mesmo offline. 1.2 Proposta de Valor Companheiro com personalidade: o pet reage com emoções, aprende hábitos e evolui com o uso. Zero atrito: interação apenas por voz — sem digitar, sem desbloquear o iPhone. Memória real: o app lembra o que você disse, guarda notas e acompanha seu progresso. Proatividade: lembretes contextuais baseados em hora, localização e padrões do usuário. Privacidade first: transcrição e LLM rodam no dispositivo, dados sensíveis não saem do watch. Extensibilidade: arquitetura preparada para sincronizar com serviços externos no futuro. 1.3 Plataforma-Alvo Componente Especificação Notas Dispositivo primário Apple Watch Series 9 / Ultra 2 / SE (2ª gen) Neural Engine 4-core no S9 Sistema Operacional watchOS 10+ SwiftUI nativo, Live Activities Companion app iOS iPhone com iOS 17+ OAuth, sincronização, configuração avançada Chip Apple S9 SiP Suporte a Core ML on-device Framework de ML Core ML 7, Create ML, Speech framework Transcrição offline Conectividade Wi-Fi, Bluetooth, LTE (cellular) Fallback offline total garantido 2. Funcionalidades e Módulos 3.1 Mapa de Funcionalidades Módulo Funcionalidades Prioridade 🎙 Voz &amp; Transcrição Ativação por levantar pulso, wake word, ASR on-device, feedback visual de áudio P0 — MVP 🐾 Pet &amp; Personalidade Avatar animado, 5 estados emocionais, personalidade configurável, evoluição com uso P0 — MVP ⏰ Lembretes Inteligentes Lembrete por voz, repetição, lembretes proativos de hidratação/postura/pausas P0 — MVP ⏱ Timer &amp; Cronômetro Criar timer por voz, timer Pomodoro, alertas hápticos, sobreposição de timers P0 — MVP 📝 Bloco de Notas Ditar notas, categorização automática por LLM, busca semântica, exportar para iPhone P1 🧠 Memória Contextual Histórico de conversas, perfil de preferências, detecção de padrões de uso P1 📊 Dashboard de Hábitos Visualização de hidratação, check-ins concluídos, streak de dias P2 🔗 Integration Layer Protocolo extensível para conectar serviços externos; fundação no MVP, conectores na v2+ P1 (arquitetura) / P2 (conectores) ⚙ Configurações Escolha de personalidade do pet, voz, idioma, horários ativos, privacidade, integrações P1 3.2 Detalhamento: Módulo de Voz e Transcrição O coração do app é o pipeline de reconhecimento de voz. Para garantir baixa latência e privacidade, a transcrição roda inteiramente no dispositivo usando o Speech framework da Apple com SFSpeechRecognizer em modo offline. O fluxo é: Detecção de ativação: acelerômetro detecta gesto de levantar pulso (WKExtendedRuntimeSession) ou crown press duplo. Captura de áudio: AVAudioSession em modo .record com cancelamento de ruído ativo disponível no S9. Transcrição contínua: SFSpeechAudioBufferRecognitionRequest com shouldReportPartialResults = true para feedback instantâneo. Detecção de pausa: silêncio de 1.2s encerra o segmento e envia ao LLM. Fallback online: se offline não reconhece, tenta via URLSession com endpoint privado do iPhone. 3.3 Detalhamento: LLM Embarcado Abordagem técnica: Usar um modelo quantizado de 200-500M parâmetros (ex: phi-2 Q4, Gemma-2B Q4, ou TinyLlama) convertido para Core ML via coremltools. O modelo é carregado uma vez na memória e processa tokens on-device com o Neural Engine. Modelo Candidato Características phi-2 (2.7B → Q4 ~1.3GB) Excelente raciocínio para o tamanho; requer Watch Ultra/Series 9 com 1GB RAM TinyLlama 1.1B Q4 (~600MB) Mais leve; cabe no Series 9 base; menor qualidade de contexto longo Gemma-2B Q4 (~1.1GB) Boa qualidade; suporte Google; requer benchmark no device Modelo proprietário fine-tuned Treinar modelo customizado focado em intenções do app (~50M params); menor, mais rápido Recomendação de MVP: iniciar com modelo proprietário fine-tuned de classificação de intenção (~50M params) + geração de respostas pré-definidas por template. Isso garante latência &lt; 300ms e cabe em qualquer Apple Watch desde o SE. O modelo generativo completo entra na v2. 4. Arquitetura de Integrações Externas Princípio central: As integrações externas NÃO fazem parte do MVP, mas a fundação arquitetural para suportá-las deve ser construída desde a Fase 1. Isso evita reescrita futura e mantém o core do app desacoplado de qualquer serviço específico. 10.1 Visão Geral do Design A Integration Layer reside exclusivamente no iPhone companion app por três razões fundamentais: o Watch tem restrições severas de rede e bateria; o OAuth exige um browser — disponível apenas no iPhone via ASWebAuthenticationSession; e o iPhone é o hub natural para coordenar sincronização em background sem drenar o Watch. O Watch apenas produz eventos (notas criadas, lembretes disparados) e os enfileira localmente. O iPhone consome essa fila, processa as integrações e confirma o sync de volta ao Watch via WatchConnectivity. Esse modelo garante que o Watch funcione perfeitamente mesmo sem iPhone nas proximidades ou sem conexão à internet. 10.2 ConnectorProtocol — O Contrato Central Toda integração externa — seja Notion, Google Calendar, Todoist ou qualquer serviço futuro — deve implementar ConnectorProtocol. Esse protocolo define o contrato mínimo que qualquer conector precisa honrar, independente das particularidades da API de destino. protocol ConnectorProtocol { var id: String { get } // ex: &quot;notion&quot;, &quot;gcal&quot; var displayName: String { get } // ex: &quot;Notion&quot;, &quot;Google Calendar&quot; var supportedEntityTypes: [EntityType] { get } // .note, .reminder, .event // Autenticação func authenticate() async throws -&gt; AuthCredential func refreshToken(_ credential: AuthCredential) async throws -&gt; AuthCredential func revokeAccess() async throws // Sincronização func push(_ entity: SyncableEntity) async throws -&gt; ExternalID func pull(since: Date) async throws -&gt; [SyncableEntity] func delete(externalID: ExternalID) async throws // Health check func validateConnection() async throws -&gt; Bool } A separação entre autenticação, push, pull e delete garante que cada conector possa ser implementado, testado e substituído de forma independente, sem afetar os demais nem o core do app. 10.3 IntegrationRegistry — Descoberta de Conectores O IntegrationRegistry é um singleton no iPhone companion app responsável por manter o catálogo de conectores disponíveis, instanciar o conector correto sob demanda e expor a lista de integrações ativas para a UI de configurações. class IntegrationRegistry { static let shared = IntegrationRegistry() // Registro de conectores disponíveis no app private var connectors: [String: ConnectorProtocol] = [ &quot;notion&quot;: NotionConnector(), &quot;gcal&quot;: GoogleCalendarConnector(), // Novos conectores são adicionados aqui — sem tocar no resto ] func connector(for id: String) -&gt; ConnectorProtocol? { return connectors[id] } func enabledConnectors(for entityType: EntityType) -&gt; [ConnectorProtocol] { return connectors.values.filter { $0.supportedEntityTypes.contains(entityType) &amp;&amp; isEnabled($0.id) } } } Adicionar um novo conector no futuro é tão simples quanto criar uma nova classe que implementa ConnectorProtocol e registrá-la no dicionário acima. Nenhuma outra parte do sistema precisa ser modificada. 10.4 SyncQueue — Resiliência a Falhas de Rede A SyncQueue é uma fila persistida em Core Data que desacopla a produção de eventos (Watch) do consumo (iPhone → API externa). Ela garante que nenhuma ação do usuário seja perdida, mesmo que o iPhone esteja sem internet, a API externa esteja fora do ar, ou o usuário feche o app durante a sincronização. struct SyncQueueItem { let id: UUID let entityType: EntityType // .note, .reminder, .event let entityID: UUID // ID local no Core Data let operation: SyncOperation // .create, .update, .delete let connectorID: String // &quot;notion&quot;, &quot;gcal&quot;, ... var status: SyncStatus // .pending, .inProgress, .done, .failed var retryCount: Int // máx. 5 tentativas var nextRetryAt: Date? // backoff exponencial: 30s, 2min, 10min, 1h, 24h let createdAt: Date } O SyncEngine processa a fila em background usando BGProcessingTask (iOS Background Tasks framework). Em caso de falha, aplica backoff exponencial com limite de 5 tentativas antes de marcar o item como .failed e notificar o usuário discretamente via complication ou notificação no Watch. 10.5 Fluxo Completo de Sincronização O diagrama abaixo descreve o caminho completo desde a ação do usuário no Watch até a persistência no serviço externo: Etapa Componente Descrição 1. Ação no Watch NoteManager / ReminderEngine Usuário cria nota ou lembrete por voz; entidade salva em Core Data local no Watch 2. Enfileiramento local Watch: SyncQueueItem Item criado na fila local do Watch com status .pending e connectorID do destino 3. Transferência Watch→iPhone WatchConnectivity (transferUserInfo) Fila transferida de forma confiável e assíncrona para o iPhone; sem perda de dados 4. Recepção no iPhone AppDelegate / WCSessionDelegate iPhone recebe o item, persiste em seu próprio Core Data e dispara o SyncEngine 5. Autenticação OAuthManager + Keychain Token recuperado do Keychain; se expirado, refresh automático via ConnectorProtocol.refreshToken 6. Push para API ConnectorProtocol.push() Conector específico (NotionConnector, GoogleCalendarConnector) chama a API com retry 7. Confirmação de ID externo Note.externalIDs ID retornado pela API externa salvo no campo externalIDs da entidade local 8. Confirmação ao Watch WatchConnectivity (sendMessage) iPhone notifica o Watch que o sync foi concluído; status atualizado para .done 9. Feedback ao usuário Pet animation / Complication Pet exibe animação de &apos;sincronizando&apos; discreta; complication atualiza ícone de status 10.6 Gerenciamento de Autenticação (OAuth 2.0) Toda autenticação de serviços externos usa OAuth 2.0 via ASWebAuthenticationSession no iPhone. O usuário nunca vê uma tela de login no Watch — o fluxo completo acontece no iPhone e o Watch é notificado do resultado. class OAuthManager { // Inicia fluxo OAuth no iPhone — apresenta SafariVC func authorize(connector: ConnectorProtocol) async throws -&gt; AuthCredential // Persiste tokens de forma segura no Keychain func save(_ credential: AuthCredential, for connectorID: String) // Recupera e renova token automaticamente se necessário func validCredential(for connectorID: String) async throws -&gt; AuthCredential // Revoga acesso e remove tokens do Keychain func revoke(connectorID: String) async throws } Segurança: Tokens de acesso e refresh são armazenados exclusivamente no Keychain do iPhone com proteção kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly. Nunca são persistidos em Core Data, logs ou transmitidos ao Watch. 10.7 ConflictResolver — Sincronização Bidirecional Quando um conector suporta pull (ex: Google Calendar), existe a possibilidade de conflito entre a versão local e a versão remota de uma entidade. O ConflictResolver aplica a seguinte política por padrão, configurável por tipo de entidade: Situação Política padrão Configurável? Entidade modificada localmente e remotamente Last-write-wins baseado em updatedAt Sim — pode ser &apos;local sempre ganha&apos; ou &apos;remoto sempre ganha&apos; Entidade deletada localmente, existe remotamente Deleta remotamente Sim — pode ser &apos;manter remoto&apos; Entidade criada remotamente, não existe localmente Importa localmente com tag de origem Não — sempre importa Conflito irresolvível (timestamps iguais) Cria duplicata com sufixo &apos;(conflito)&apos; Não — sempre notifica usuário 10.8 Conectores Planejados — Roadmap de Integrações Conector Entidades suportadas Fase planejada Notion Notes → Notion Pages, Reminders → Notion Tasks (Database) Fase 4 Google Calendar Reminders/Timers → GCal Events, pull de eventos do dia Fase 4 Apple Notes Notes → Apple Notes (via NSItemProvider / Shortcuts) Fase 4 (nativo, sem OAuth) Todoist Reminders → Todoist Tasks, pull de tarefas pendentes Fase 5 Obsidian Notes → Markdown files via Obsidian URI scheme Fase 5 Slack Notes → Slack messages (para si mesmo via DM) Fase 5 Webhook genérico Qualquer entidade → POST para URL customizada Fase 6 (power users / devs) Critério de priorização: Notion e Google Calendar foram priorizados por serem os mais requisitados pelo perfil de personas definido. Apple Notes entra junto por não exigir OAuth — é uma integração local via Shortcuts que pode ser implementada em dias, não semanas. 10.9 UI de Configuração de Integrações (iPhone companion app) A tela de Integrações no iPhone companion app é o único ponto de configuração. O Watch não tem tela de configuração de integrações — apenas exibe status. Lista de conectores disponíveis com status (Conectado / Não conectado / Erro de autenticação). Toggle por conector: ativar/desativar sincronização sem desconectar a conta. Configuração por entidade: usuário escolhe quais tipos sincronizar (ex: Notion só para notas, não para lembretes). Status de sync: última sincronização bem-sucedida, itens pendentes na fila, erros recentes. Botão &apos;Sincronizar agora&apos; para sync manual on-demand. Seção &apos;Avançado&apos;: política de conflito por conector, frequência de pull, log de eventos de sync. 10.10 O que NÃO fazer — Anti-patterns de Integração Atenção: Os erros abaixo são comuns em apps com integrações externas e devem ser explicitamente evitados no WatchPet. Nunca fazer chamadas de rede diretas do Watch para APIs externas. O Watch não tem gerenciamento eficiente de background network e drena bateria rapidamente. Nunca bloquear a UX do Watch aguardando confirmação de sync. O Watch registra a ação e segue em frente; o sync é sempre assíncrono e transparente. Nunca armazenar tokens OAuth fora do Keychain. Core Data, UserDefaults e logs são vetores de vazamento de credenciais. Nunca acoplar o Core Domain ao ConnectorProtocol. NoteManager não sabe que Notion existe — ele apenas emite eventos; a Integration Layer os consome. Nunca sincronizar sem idempotência. Cada item da SyncQueue deve ter um ID único; reprocessar o mesmo item não deve criar duplicatas. WatchPet — AyD v2.0 • Documento gerado com assistência de IA
